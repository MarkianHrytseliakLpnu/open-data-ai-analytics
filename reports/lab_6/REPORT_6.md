# Звіт до лабораторної роботи №3

## 1. Огляд проєкту
Метою даної лабораторної роботи був перехід від монолітного локального виконання скриптів до мікросервісної архітектури за допомогою контейнеризації. Проєкт аналітики відкритих даних було повністю контейнеризовано з використанням **Docker** та **Docker Compose**. Це забезпечило ізольованість середовищ, незалежність модулів один від одного та легкість розгортання проєкту на будь-якій машині без необхідності попереднього налаштування локального Python-середовища.

## 2. Архітектура та створені сервіси

Проєкт розділено на окремі контейнери, кожен з яких базується на власному оптимізованому `Dockerfile` (використано легкий образ `python:3.11-slim` зі встановленням лише необхідних бібліотек). 

Загалом створено **7 сервісів**, які поділяються на дві функціональні групи:

### Інфраструктурні сервіси (Long-running)
Ці сервіси працюють постійно у фоновому режимі:
* **`db` (PostgreSQL):** Головна база даних, яка зберігає очищені та структуровані дані.
* **`pgadmin`:** Веб-інтерфейс для візуального адміністрування БД та виконання SQL-запитів через браузер.
* **`web` (Streamlit App):** Веб-дашборд (`app.py`), що відображає завантажені дані, звіти у форматі JSON та згенеровані графіки.

### Аналітичні сервіси (Batch Jobs)
Ці сервіси виконують конкретне завдання та завершують роботу після його виконання:
* **`loader`:** Завантажує CSV-файл з порталу [data.gov.ua](https://data.gov.ua/) та імпортує його до PostgreSQL.
* **`quality`:** Аналізує датасет на наявність пропусків, дублікатів та помилок форматування.
* **`research`:** Обчислює статистичні показники та знаходить ключові інсайти.
* **`visualization`:** Створює візуалізації (гістограми, діаграми, теплокарти) у форматі PNG.

## 3. Організація взаємодії між сервісами

Взаємодія між ізольованими контейнерами організована через внутрішню мережу та систему спільних томів (volumes).

### 3.1. Файлова взаємодія (Docker Volumes)
Для передачі артефактів між етапами пайплайну використано спільні томи:
* **`shared_data`:** Дозволяє модулю `loader` зберегти завантажений файл, а іншим модулям — прочитати його для аналізу.
* **`shared_reports`:** Використовується для зберігання JSON-звітів та графіків. Це дозволяє сервісу `web` миттєво відображати результати роботи аналітичних модулів.
* **`postgres_data`:** Забезпечує персистентність (збереження) даних PostgreSQL навіть після видалення контейнерів.

### 3.2. Мережева взаємодія (Docker Network)
Усі сервіси об'єднані в одну віртуальну мережу `analytics_net`.
* Модуль `loader` підключається до бази даних за внутрішнім ім'ям хоста (`db`) замість `localhost`.
* Назовні (для користувача) прокинуто лише порти веб-інтерфейсів: `8501` (Streamlit) та `5050` (pgAdmin).

### 3.3. Оркестрація та послідовність
Для забезпечення правильного порядку виконання використано директиву `depends_on`:
1. `loader` запускається тільки після успішного проходження `healthcheck` базою даних.
2. Модулі `quality`, `research` та `visualization` запускаються лише після успішного завершення роботи сервісу `loader`.

## 4. Вирішення проблем під час налаштування (Troubleshooting)

Під час роботи над контейнеризацією було виявлено та вирішено такі труднощі:

1.  **Помилка доступу до БД через браузер (`ERR_CONNECTION_RESET`):**
    * *Проблема:* Спроба відкрити порт `5432` безпосередньо в браузері викликала помилку, оскільки PostgreSQL не підтримує HTTP.
    * *Рішення:* Додано сервіс `pgadmin`, який виступає веб-клієнтом для бази даних на окремому порту `5050`.
2.  **Конфлікт абсолютних шляхів у тестах:**
    * *Проблема:* Модулі використовували шлях `PROJECT_ROOT` з `config.py`, через що тести намагалися писати файли в реальну папку проєкту замість тимчасової папки Pytest.
    * *Рішення:* У тестах використано `@patch('src.quality.config')` та `@patch('src.research.config')` для динамічного перенаправлення шляхів звітів у тимчасові папки під час тестування.
3.  **Ініціалізація веб-інтерфейсу:**
    * *Проблема:* Streamlit намагався завантажити графіки, які ще не були створені модулем `visualization`.
    * *Рішення:* В `app.py` додано перевірки `if path.exists()` перед завантаженням JSON та PNG, щоб уникнути падіння інтерфейсу та показувати користувачу інформативні повідомлення про стан виконання пайплайну.

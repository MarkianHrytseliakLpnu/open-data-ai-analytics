# Звіт до лабораторної роботи №2

## 1. Огляд проєкту
Метою даної лабораторної роботи було налаштування автоматизованих конвеєрів безперервної інтеграції та розгортання (CI/CD) для проєкту аналітики відкритих даних. Було реалізовано два підходи: використання хмарних серверів (GitHub-hosted) та налаштування локального сервера (Self-hosted runner) на базі ОС Windows. Також було імплементовано автоматичну генерацію та публікацію результатів аналізу (графіків) на GitHub Pages.

## 2. Структура Workflow (Етапи CI/CD)

Роботу конвеєрів розділено на два окремі workflow-файли для різних середовищ виконання:

### Хмарний конвеєр (GitHub-hosted CI/CD)
- **Файл:** `.github/workflows/ci.yml`
- **Функціонал:** - **Detect Changes:** Використовує дію `dorny/paths-filter` для визначення змінених файлів у репозиторії.
  - **CI (Матричне тестування):** Запускає модулі (`loader`, `quality`, `research`, `visualization`) паралельно на віртуальних машинах `ubuntu-latest`. Завдяки умові `if` тестуються лише ті модулі, файли яких були змінені (у випадку Pull Request), або всі (при ручному запуску чи Push у `main`).
  - **CD (Deploy Pages):** При успішному проходженні тестів у гілці `main`, запускає повний пайплайн, генерує HTML-звіт з отриманими фігурами та розгортає його на GitHub Pages.

### Гібридний/Локальний конвеєр (Self-hosted CI/CD)
- **Файл:** `.github/workflows/ci-selfhosted.yml`
- **Функціонал:**
  - **Локальне тестування:** Виконується на локальному ПК (Windows) за допомогою тегу `runs-on: self-hosted`. Реалізовано ручний запуск (`workflow_dispatch`) з можливістю вибору конкретного модуля через інтерфейс GitHub.
  - **Гібридний CD:** Після успішного локального тестування, етап деплою передається на хмарні сервери GitHub (`ubuntu-latest`). Це дозволяє уникнути проблем із сумісністю Linux/Windows bash-скриптів під час генерації та публікації статичного сайту.

## 3. Вирішення проблем під час налаштування (Troubleshooting)

У процесі налаштування CI/CD було виявлено та вирішено низку специфічних проблем:

1. **Проблема області видимості змінних матриці (Matrix Scope Issue):**
   - *Опис:* GitHub Actions оцінює умови `if` на рівні `jobs` до ініціалізації матриці, що викликало помилку `Unrecognized named-value: 'matrix'`.
   - *Рішення:* Логіку перевірки було перенесено у глобальну змінну середовища `env: SHOULD_RUN`, а саму перевірку додано до кожного окремого кроку (`steps`).

2. **Помилка EBUSY (Resource busy or locked) на Windows Runner:**
   - *Опис:* Стандартний екшен `actions/checkout@v4` намагався видалити тимчасову папку `.gitconfig`, але системні фонові процеси (Git credential manager або процеси антивірусу) утримували її. Це призводило до фатального падіння пайплайну через стан перегонів (Race Condition).
   - *Рішення:* Було відмовлено від стандартного екшену `checkout`. Замість нього написано кастомний крок ("Bulletproof for Windows"), який завантажує код прямими командами термінала: `git clone`, `git fetch`, `git clean -fd`. Це повністю усунуло проблему блокування папок.

3. **Попередження про застарілі версії Node.js (Node 20 deprecation):**
   - *Опис:* Запуск деяких екшенів супроводжувався попередженнями про швидке припинення підтримки Node.js 20 серверами GitHub.
   - *Рішення:* Додано системну змінну середовища `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24: true` для примусового використання оновленого середовища.

## 4. Порівняння продуктивності (Self-hosted vs GitHub-hosted)

Було проведено порівняння часу виконання повного циклу тестування (всіх 4 модулів) на різних середовищах:

- **GitHub-hosted (Ubuntu):** ~1 хв 39 сек (+5 сек на етап detect changes).
- **Self-hosted (Local Windows):** ~2 хв 37 сек.

**Пояснення результатів:**
Незважаючи на те, що виконання окремих тестів на локальному комп'ютері відбувається приблизно на 10 секунд швидше (завдяки відсутності необхідності щоразу завантажувати контейнери та встановлювати залежності з нуля), загальний час на GitHub-hosted виявився суттєво меншим.

Це зумовлено **архітектурою паралелізації**. При використанні стратегії матриці (`strategy: matrix`), сервери GitHub під капотом виділяють 4 окремі віртуальні машини, які виконують тестування модулів *одночасно*. Натомість локальний Self-hosted раннер працює як єдиний вузол. Отримавши 4 завдання з матриці, він ставить їх у чергу і виконує *послідовно* одне за одним, що збільшує загальний час виконання конвеєра майже вдвічі.
